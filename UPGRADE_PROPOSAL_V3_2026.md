## 1. 텔레그램 차트 보고서 자동 발송

### 문제
320+ API 엔드포인트, 25개 DB 테이블의 데이터가 있지만, 원장님은 대시보드를 매일 열어보지 않음. **폰에서 30초 내에 파악 가능한 보고서**가 필요.

### 구현

**주간 보고서 (매주 월요일 09:00)**:
1. matplotlib로 차트 이미지 생성 (순위 추이, 리뷰 현황, 키워드 성과)
2. Gemini로 한국어 내러티브 요약 생성 ("이번 주 '청주 한의원' 15위→12위 상승")
3. 텔레그램 sendPhoto로 차트 + 텍스트 전송

**핵심 알림 임계값**:
| 알림 | 임계값 | 우선순위 |
|------|--------|---------|
| 순위 5위+ 하락 | 즉시 | CRITICAL (하루 최대 3건) |
| 순위 3위+ 하락 | 주간 다이제스트 | WARNING |
| TOP 3 진입 | 즉시 | CRITICAL (축하) |
| 경쟁사 리뷰 20건+/일 | 즉시 | CRITICAL |
| 7일간 리뷰 0건 | 주간 다이제스트 | WARNING |

**의존성**: `kaleido>=0.2.1` (plotly 이미지 렌더링), `matplotlib` (이미 설치됨)

## 2. 스크래퍼 헬스 모니터

### 문제
14개 스크래퍼가 매일 실행되지만, DOM 변경/차단/오류로 깨져도 즉시 알 수 없음. `data_health_monitor.py`가 있지만 수집량 통계만 추적.

### 구현

**각 스크래퍼 실행 시 메트릭 자동 수집**:
```python
@dataclass
class ScraperMetrics:
    records_collected: int     # 수집 건수
    empty_fields_ratio: float  # 빈 필드 비율 (0.0~1.0)
    error_count: int           # 에러 횟수
    duration_seconds: float    # 실행 시간
    unique_ratio: float        # 신규 vs 중복 비율
```

**z-score 기반 이상 감지**:
- 14일 이동 평균 대비 현재 수집량이 2σ 이하 → WARNING
- 3σ 이하 또는 0건 → CRITICAL (스크래퍼 깨짐 판정)

**자동 복구 패턴**:
- CSS 셀렉터 폴백 체인 (주 셀렉터 실패 시 대안 셀렉터 시도)
- 체크포인트/재개 (중단 시 처음부터 다시 시작하지 않음)

## 3. 전환 추적 (CPA/ROI 측정)

### 문제
마케팅 비용을 쓰지만 **어떤 채널이 실제로 환자를 데려오는지** 모름.

### 구현 - Google Sheets Micro-CRM 연동

**네이버 스마트콜 연동**:
- 무료로 050 번호 5개 할당 가능 (채널별 1개)
- 채널별 전화 유입 자동 추적

**접수 시 유입경로 자동 집계**:
```
DB 테이블: patient_attribution
- date, source_channel, patient_type (new/return)
- treatment_type, revenue, coupon_code
```

**CPA 자동 계산 API**:
```
월별 CPA = 채널 마케팅 비용 / 채널에서 온 신환 수
월별 ROAS = 채널에서 온 환자 매출 / 채널 마케팅 비용
```

**오프라인→온라인 연결**:
- 접수 문진표에 "어떻게 알게 되셨나요?" 필수 항목
- 채널별 쿠폰 코드 (BLOG0326, INSTA0326, FRIEND)
- 오프라인 전단지/명함에 QR 코드 (스캔 추적)

## 4. 스마트플레이스 소식 자동 발행 + Q&A 모니터

### 문제
네이버 플레이스 "소식" 탭 업데이트가 **최신성 점수에 직접 영향**. 하지만 수동으로 관리해야 함. 공식 API 없음.

### 핵심 발견
- **소식 자동 발행 API: 없음** → Playwright 브라우저 자동화만 가능
- **Q&A 자동 모니터: 없음** → 완전히 미개척 영역 (도구 0개)
- **블로그 예약 발행**: metaWeblogAPI (OAuth) 또는 Playwright
- **방문자리뷰 > 블로그리뷰**: 2025-2026 알고리즘에서 방문자리뷰 가중치 상승

### 구현

**소식 자동 발행** (Playwright 기반):
1. `new.smartplace.naver.com` 로그인 (쿠키 세션 유지)
2. 소식 작성 → 텍스트 + 사진 업로드 → 발행
3. 시즌 캠페인(V2-5)과 연동하여 자동 소식 생성
4. 주 2-3회 자동 발행 → 최신성 점수 유지

**Q&A 모니터** (Playwright 스크래핑):
1. Place 페이지에서 Q&A 섹션 주기적 스캔
2. 새 질문 감지 → 텔레그램 알림 + AI 응답 초안 생성
3. 인라인 키보드(V2-2)로 원장님 승인 → 자동 답변 게시

**MY플레이스 저장 유도**:
- 소식 발행 시 "MY플레이스에 저장해두세요!" CTA 삽입
- 리뷰 요청 시 저장 동시 유도

## 5. 추가 실전 개선안

### 5-1. 데이터 보존 정책 자동화
- 90일 이상 된 raw 데이터 → `db/archive/` 백업 후 삭제
- rank_history, community_mentions 등 누적 테이블 롤링 윈도우
- monthly VACUUM 자동 실행 (DB 크기 최적화)

### 5-2. 스크래퍼 적응형 스케줄링
- 변경 빈도가 낮은 소스는 주 1회로 축소 (예: HIRA 의료기관)
- 변경 빈도가 높은 소스는 일 2회로 확대 (예: 커뮤니티 멘션)
- 실행 이력 분석 → 최적 스케줄 자동 제안

### 5-3. 네이버 Place 사진 관리
- 경쟁사 Place 사진 주기적 스크린샷 → 시각 변경 감지
- 사진 품질 분석 (해상도, 밝기, 구도) → 교체 제안
- EXIF 메타데이터에 위치 정보 자동 삽입

### 5-4. 블로그 예약 발행 파이프라인
- V2-8 블로그 파이프라인 → Playwright로 네이버 블로그 자동 게시
- 주의: AuthGR AI 스팸 감지 → 반드시 인간 편집 후 게시
- metaWeblogAPI 연동 (OAuth 2.0, 블로그별 활성화 필요)
